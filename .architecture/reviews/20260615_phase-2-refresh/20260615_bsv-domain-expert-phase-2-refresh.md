# Specialist Review — Phase 2 refresh classification (issue #290)

**Reviewer:** James Thornton, BSV Domain Expert (`domain_expert`)
**Target:** Comment "Phase 2 refresh — classification after #307 + #296 egress landed (2026-06-15)" on issue #290 (umbrella #291)
**Date:** 2026-06-15
**Scope:** BRC-100 / BSV protocol correctness of the proposed `Engine::Action` decomposition — design only, no code in this HLR.

## Perspective

I evaluate every boundary against what BRC-100 requires and what the BSV protocol permits. For this refresh that means three things: does the SPV trust model survive the proposed collapse of verification into one `verify_beef` adapter; does BRC-42 change derivation stay correct when it moves to `ChangeGenerator`; and do the BRC-95 internalize/import paths and the #183 4-phase lifecycle invariants survive being scattered across `BeefImporter` / `Hydrator` / `TxBuilder`. Classification refactors are where protocol invariants quietly erode, because the methods read as "machinery" right up until you notice one of them is the only thing enforcing a consensus or SPV rule.

## Assessment

**Approve-with-changes.** The classification is protocol-sound and the two corrections are correct. Correction 1 (verification is SDK-delegated, not a collaborator) is exactly right and removes a phantom responsibility. Correction 2 (promotion-as-a-row unblocks #60) is accurate. My changes are guardrails to be written into the extraction sub-HLRs' acceptance criteria, not objections to the classification itself — three of them are load-bearing for SPV/consensus correctness and must not be lost in the move.

## Strengths

- **The trust asymmetry is named, not collapsed.** The refresh is explicit that the two adapters differ in *which chain-tracker* they pass — network-backed `ChainTracker` (incoming, untrusted) vs `TrustedSelfChainTracker` (own egress, structural-only). This is the substance of ADR-015 (egress) and the asymmetry survives the de-dup *provided* the helper keeps the tracker as a required, caller-supplied argument (see Concern 1). Folding two ~6-line adapters that already delegate to `Transaction::Tx#verify` into one shared helper is correct per ADR-018: `verify` is the stateless SDK operation, the tracker is the wallet's injected state. There is no verification *logic* to extract — only parse/store and tracker-choice, which is what `BeefImporter` rightly keeps.
- **`build_atomic_beef` / `wire_ancestor` → `Hydrator` is the right cut, and the ordering constraint is correct.** BEEF graph reconstruction is genuinely not action behaviour — `wire_ancestor` reads only the ProofStore, nothing off `@row`. Sequencing Hydrator before BeefImporter (Phase 5 → 6) is protocol-necessary: `hydrate_known_sources!` (the trustSelf BRC-95 path) calls `wire_ancestor` to fill TXID-only ancestor gaps before verify can walk the graph. BeefImporter consumes the primitive; it must exist first.
- **Promotion-as-a-row genuinely unblocks #60.** With `outputs.promoted` gone and authorisation enforced by the `promotions` composite FKs (ADR-023/019), `resolve_internalize_output` no longer needs the `output_type = 'root'` inference as a load-bearing guess — the schema now rejects an unauthorised promotion structurally. Carrying #60's engine-side half (state `output_type` explicitly) into the BeefImporter extraction rather than relocating the inference verbatim is the correct call and aligns with ADR-010's inference ban.

## Concerns

### Concern 1 — `verify_beef` must keep the chain-tracker a required argument, never a default (severity: high)

The proposed signature `verify_beef(tx, chain_tracker:, error:)` is correct *only* if `chain_tracker:` has no default. `TrustedSelfChainTracker` says "yes" to every merkle-root lookup (`valid_root_for_height? => true`); its own class comment forbids use on incoming data, because applied to an untrusted peer BEEF it would accept forged merkle proofs (ADR-015 Consequences/Negative names this the footgun). If the de-dup introduces a default tracker — or if a future caller omits the argument and the helper falls back to the trusted one — the wallet silently downgrades incoming SPV to structural-only and the BRC-95 internalize path accepts forged proofs. **Fix:** the sub-HLR acceptance criteria must state that `verify_beef` takes `chain_tracker:` with no default; that `verify_incoming_transaction!`'s callsite passes `@engine.chain_tracker` (the network-backed tracker) and still hard-raises `InvalidBeefError` when `chain_tracker` is nil (the guard at `action.rb:1094` must move with it, not be dropped); and that `validate_for_handoff!`'s callsite passes `TrustedSelfChainTracker.new`. A spec must prove an incoming-BEEF verify with the trusted tracker is *unreachable* through the public surface.

### Concern 2 — the egress structural-completeness assertion must not be optimised away as "redundant verify" (severity: high)

`validate_for_handoff!` and `verify_incoming_transaction!` look superficially like the same `verify` call with a different tracker, which invites a maintainer to treat egress validation as belabouring a check the SDK already did. It is not redundant: it is the SPV honesty contract (ADR-015 decision (a)) — the wallet refuses to *emit* a BEEF whose ancestry does not close, catching the exact multi-hop fault that motivated #296 (a proven ancestor shipped without its merkle_path). The egress assertion is a standing invariant, not a duplicate of incoming verify. **Fix:** the sub-HLR must keep egress validation wired into *both* emission points (`Action.create` synchronous path and `Action#sign!`) after the de-dup — the helper changes the implementation, not the two callsites — and must raise `EgressBeefInvalidError` (not the incoming `InvalidBeefError`) on the egress path. The distinct error class is part of the contract: egress failure means an upstream proof-closure bug, not bad peer data. Note this invariant lives in application code with no schema backstop (ADR-015 flagged deviation from ADR-003, routed to #296 Phase D); the extraction must not erode it further by making it skippable.

### Concern 3 — BRC-42 change derivation in `ChangeGenerator` must preserve the sign-after-shuffle-after-fee ordering and the `counterparty: 'self'` derivation (severity: medium)

`generate_change` is not a pure "make change outputs" utility — it encodes consensus-critical ordering (`action.rb:909-1018`): build → attach P2PKH templates → explicit `compute_fee` shortfall check → `distribute_change` → shuffle → **sign last**, because the sighash commits to final output values *and* positions. If `ChangeGenerator` is extracted as a value-returning helper that hands change specs back to `TxBuilder` for assembly/signing, the shuffle-then-sign ordering can be split across two collaborators and silently reordered, producing transactions whose signatures do not match their final output set — invalid by consensus, rejected at broadcast. Equally, the BRC-42 self-derivation (`protocol_id: [2, prefix]`, `key_id: suffix`, `counterparty: 'self'`, `derive_public_key` → `hash160` → `p2pkh_lock`) and the recording of `sender_identity_key: identity_key` on each change spec must travel intact — these are what make the change output recoverable and re-derivable later (the WBIKD recoverability property). **Fix:** the Phase 4 sub-HLR must specify that fee-distribution, shuffle, and sign stay in one collaborator (whichever owns final assembly), and that `ChangeGenerator`'s output is the *derived change templates* (pre-fee, satoshis=0, `change: true`) plus the derivation triple — not signed outputs. A spec must assert that a randomised multi-output change tx verifies after construction (it already must, but the boundary makes regression easy).

### Concern 4 — the trustSelf ordering invariants must be documented as `BeefImporter` contract, not left as inline comments (severity: low)

The BRC-95 trustSelf path has three ordering constraints currently held only by comments in `Action.internalize` (`action.rb:252-264`): `save_beef_proofs` runs *before* `replace_known_ancestors!` (else proofs for known-but-unstored ancestors are lost to TXID-only conversion); `replace_known_ancestors!` runs *after* verify (the full graph was already validated); and `make_txid_only` does not invalidate the in-memory `source_transaction` pointers verify already walked. When these methods move to `BeefImporter` and `internalize` becomes a thin shell orchestrating them, the ordering is no longer co-located with the orchestration. **Fix:** the BeefImporter sub-HLR should carry these three invariants into the collaborator's class/method docs and add (or preserve) a spec that fails if `replace_known_ancestors!` runs before `save_beef_proofs`. Low severity because the current code is correct and the comments are good; the risk is purely that a future edit to the now-split orchestration reintroduces the bug the comments warn against.

## Recommendations

1. **Write Concerns 1–3 into the relevant sub-HLR acceptance criteria verbatim** — they are the protocol guardrails that a "mechanical extraction" can lose. Concern 1 (no default tracker) and Concern 2 (egress assertion stays wired into both emission points, distinct error class) protect the SPV trust model; Concern 3 (ordering + BRC-42 triple) protects consensus validity of change.
2. **Keep the `apply_spends` judgement call on `Action`, not `TxBuilder`.** The #290 body flags it as a toss-up. It reads `@row[:raw_tx]`, resolves inputs for the *specific* action, and completes the deferred-signing lifecycle (BRC-100 `signAction`) — it is action lifecycle behaviour that happens to do protocol work, the inverse of the "machinery that happens to run during creation" test. Leaving it on `Action` keeps the deferred-sign lifecycle legible.
3. **Have the FundingStrategy sub-HLR (Phase 3) preserve the auto-fund headroom semantics (ADR-013) explicitly** — `run_funding_loop` is bounded by `spendable_count + 1`, raises `InsufficientFundsError` on caller-supplied shortfall (no top-up), and wraps `PoolDepletedError`. These are not arbitrary; they are the auto-fund contract. Not a protocol concern per se, but the biggest-mass move and the one most likely to drift.
4. **Confirm the #192 return path is unblocked by the cut, as #291 requires.** The classification keeps the 4-phase Action intact and pushes collaborators below it, which is the right shape for #192's `Engine::Batch` to compose over — no concern here, but the extraction specs should assert the single-tx 4-phase invariants (#183: outputs canonical only post-broadcast) still hold per collaborator, so batching layers on rather than forcing parallel architecture.

## Summary line

Protocol-sound classification; both corrections correct. Approve-with-changes: the verification de-dup is safe *iff* the chain-tracker stays a required argument (no default), the egress structural-completeness assertion stays wired into both emission points with its distinct `EgressBeefInvalidError`, and BRC-42 change derivation keeps fee→shuffle→sign in one collaborator with the derivation triple intact. Fold these into the extraction sub-HLRs' acceptance criteria.
