# Extract Engine::BRC100 (#364) — Phase 7, final #291 extraction

The fifth and final extraction of the #291 "Monolith to Manageable" roadmap, after #323 (FundingStrategy), #336 (TxBuilder), #343 (Hydrator), #357 (BeefImporter). When this merges, **Monolith → Manageable is done.**

This is a different *kind* of move from the four before it. Those were genuine collaborators — DI'd objects with their own contracts and `@engine.x → injected x` rewrites. This is a **mixin facade**: the 28 BRC-100 spec methods are the public interface, not a domain. A collaborator object would force circular delegation (`engine.create_action → @brc100.create_action → engine.@store`). `Engine::BRC100` is a *slice of Engine*, not an island.

## Conceptual model

Three layers are now visible (the payoff of #291), though only the first boundary is drawn in this slice:

1. **Interface layer** (`Engine::BRC100`) — translates BRC-100 in/out: parse spec-shaped input → call engine machinery → shape BRC-100 JSON. *An* interface over the engine; currently the only one, comprehensive, tightly coupled, but in theory one of several.
2. **Machinery layer** (`Engine`) — interface-agnostic wallet operations (`build_action`, `broadcast_or_defer`, `reject_action`). The wallet *is* the database record of what we can spend; this layer orchestrates that substrate without knowing about BRC-100.
3. **Implementation layer** (`Action` + logical models) — the DB-level mutations.

This slice exposes the interface↔machinery seam **without yet separating them** (see Deferred).

## Current implementation (master)

The 28 spec methods are defined directly in `class Engine` (`engine.rb`), contiguous from `create_action` (~:119) to `get_version` (~:1030), grouped by the six BRC-100 functional areas:

- Transaction Operations (codes 1–7): `create_action`, `sign_action`, `abort_action`, `list_actions`, `internalize_action`, `list_outputs`, `relinquish_output`
- Public Key Management (8–10): `get_public_key`, `reveal_counterparty_key_linkage`, `reveal_specific_key_linkage`
- Cryptography (11–16): `encrypt`, `decrypt`, `create_hmac`, `verify_hmac`, `create_signature`, `verify_signature`
- Identity & Certificate (17–22): `acquire_certificate`, `list_certificates`, `prove_certificate`, `relinquish_certificate`, `discover_by_identity_key`, `discover_by_attributes`
- Authentication (23–24): `authenticated?`, `wait_for_authentication`
- Blockchain & Network (25–28): `get_height`, `get_header_for_height`, `get_network`, `get_version`

`reject_action` (~:195) sits physically inside the codes-1–7 section but is **not** a spec method (its own comment says so; called by bin/ tools + the daemon resolution path, not by any BRC-100 method).

SDK `Interface::BRC100` is included at `engine.rb:30` — the contract: 28 methods that `raise NotImplementedError`. Engine overrides every one. Post-phases-3–6 the impls are mostly thin (`create_action` → `validate_* + Action.create(engine: self)`; `list_actions` → `Action.list`).

## Target design

- New `gem/bsv-wallet/lib/bsv/wallet/engine/brc100.rb` = `module BSV::Wallet::Engine::BRC100`, holding the 28 method bodies **verbatim**.
- The module does `include BSV::Wallet::Interface::BRC100`.
- `Engine` does `include Engine::BRC100` and **drops** its direct `include BSV::Wallet::Interface::BRC100` (inherited transitively).
- Ancestry: `Engine → Engine::BRC100 → Interface::BRC100` — impls beat stubs.
- `engine.rb` → ~850–950 LOC.

## What stays on Engine

- Runtime container: `initialize`, `limp_mode?`, `headroom`
- Bootstrap: `import_utxo`
- Porcelain: `import_wallet`, `generate_receive_address`, `list_receive_addresses`, `scan_receive_addresses`, `send_payment`, `consolidate_step`, `sweep`, `sweep_to_root`, `estimate_sweep_fee`
- `reject_action` — machinery, not spec (fork C)
- All ~20 privates (`validate_*`, `enforce_headroom!`, `enforce_limp_mode!`, `attach_labels`, `determine_broadcast`, `publish_beef_hint`, `require_key_deriver!`, the wbikd helpers, `fetch_*`, `secure_compare`, …) — shared between moved spec methods and staying porcelain, so they stay (fork B)

## What changes

Nothing at runtime. Pure file/module split. Because it's a mixin, `self` is still the Engine instance — `@store`, `@key_deriver`, the collaborators, and every private resolve unchanged. No DI, no `engine.send`. Existing engine + integration specs are the regression guard.

## Hardest aspect / surprises

- **Method-resolution order.** The impls must beat the SDK contract's `NotImplementedError` stubs. Arranging `Engine::BRC100 include Interface::BRC100` and `Engine include Engine::BRC100` gives `Engine → Engine::BRC100 → Interface::BRC100`, which is correct. A dedicated spec locks `Engine.ancestors` order *and* asserts each of the 28 resolves to the impl, so a future reorder can't silently re-expose a stub.
- **The slice is not self-contained.** It calls back to Engine privates and reaches Engine ivars/collaborators. That's expected for a mixin facade — do not try to make it standalone (that's the deferred machinery-layer work, not this).
- **Exactly the 28.** `reject_action` is the trap — it lives inside the codes-1–7 section but does not move. Move spec methods only.
- **`require` / autoload ordering.** Ensure `engine/brc100.rb` is required before the `class Engine` body runs `include Engine::BRC100` (mirror how the other `engine/*.rb` collaborators are required).

## Deferred — captured, NOT done here

The interface↔machinery separation (the north star). Recorded on #291 (comment 4725832110):

- BRC-100 methods becoming pure translation: `engine.build_action(...) + broadcast_or_defer(...) + → json`, with `engine.build_action` calling `Action.create`.
- Interface-input validations (`validate_description!`, `validate_output_ownership!`) potentially migrating *into* the slice.
- The transitional `Action.create(engine: self)` calls inside the moved methods are a waypoint, not the destination.

This is the next roadmap phase ("Manageable → Machined") — its own analysis → classification → refactor round, seeded by what this slice exposes. No machinery extraction in this PR.

## Implementation steps (ordered)

1. Create `engine/brc100.rb` with `module Engine::BRC100` that `include`s `Interface::BRC100`. Move the 28 method bodies verbatim out of `engine.rb` into it. Leave `reject_action` and everything else on Engine.
2. In `engine.rb`: add `require_relative 'engine/brc100'` (matching the existing collaborator require pattern), replace `include BSV::Wallet::Interface::BRC100` with `include Engine::BRC100`.
3. Add MRO spec (ancestors order + each of the 28 resolves to impl).
4. Full suite (Postgres + SQLite) + rubocop, from `gem/bsv-wallet`. Dead-code / LOC sweep.

## Specs

- New MRO spec: `Engine.ancestors.index(Engine::BRC100) < Engine.ancestors.index(Interface::BRC100)`; for each of the 28, `Engine.instance_method(:m).owner == Engine::BRC100` (or the impl, not `Interface::BRC100`), and a smoke that calling one does not raise `NotImplementedError`.
- Existing engine + integration specs unchanged and green (behaviour preservation — they exercise the public surface, which is unchanged).
