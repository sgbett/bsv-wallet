# BSV Vault — feasibility-gated build mechanism

**Status:** proposed — spike-gated, nothing committed to build until the spike returns
**Origin:** session comparing `bsv-blockchain/vault-manager` against wallet-toolbox and our wallet
**Date:** 2026-06-11

## Why

`vault-manager` (Peer-to-Peer Privacy Systems Research) is an air-gapped cold-storage
signing appliance: two static React bundles plus a large operator manual. Its substance is
a *business process* — ceremony, two-person rule, removable-media chain-of-custody — with
thin software that exists to enforce the procedure. Its `Vault` class implements
`ChainTracker` by **prompting a human** to attest Merkle roots; there is no network.

The question this plan answers: **could we produce something equivalent as a verified
*subset* of our own wallet + SDK, rather than a fork?** Our layered architecture already
has the seams a vault needs:

- `ChainTracker` is an abstract interface, today realised as a DB↔network write-through
  cache. A vault is the *same cache with the network fill severed* — a miss is filled by a
  human (`isValidRootForHeight`, `currentHeight`), exactly what vault-manager does.
- Broadcast/ingress (`Engine::Broadcast`, `Engine::TxProof`) is the "talking to the outside
  world" machinery a vault replaces with human-mediated media transfer (Atomic BEEF via
  QR/files).

A vault built on our **Store** inherits *schema-enforced* assurance — "invalid state is
structurally impossible" extends into the cold vault. That is a **stronger** claim than
vault-manager, whose state is in-memory arrays guarded only by procedure. That asymmetry is
the most interesting reason to build it.

The one genuine convergence with the reference is **Atomic BEEF as the mandatory custody
envelope** — independent validation of the egress-validity direction (#296).

## Core design (agreed in session)

### It is a packaging problem, not a runtime mode

Assurance is a claim about what code is **absent**, not what a flag toggles off. A feature
flag that disables network egress still ships the egress code; proving it can't be flipped
is far harder than proving it isn't there. So the security boundary is the **build**, not a
runlevel. Three assurance levels share one set of seams:

1. Runtime flag — dev ergonomics only; worthless to an auditor.
2. Composition / separate entrypoint (`bin/vault`) — code present in tree, not loaded.
3. Separate artifact — network machinery physically absent. The real boundary.

The right analogy is *a kernel compiled without the network stack*, not rc1-vs-rc3.

### "Necessary and sufficient" is asymmetric

- **Sufficiency is verifiable** — "does the subset run every vault op?" Boot it, run the
  ceremony suite, green/red.
- **Necessity is not** — Ruby's dynamic dispatch (`const_get`, `send`, autoload) defeats
  static reachability. You can only *approximate* minimality.

Tractable here **only because the vault's operation set is small and ceremonial** (ingest
BEEF, derive address, build+sign, export BEEF) — unlike the open-ended wallet surface.

### Selection is a reviewed manifest, not a live trace

A trace is non-deterministic; "rebuild and verify before install" needs determinism. So:
**the trace is a dev-time tool that derives the manifest; the manifest is what ships.** The
manifest (explicit file list + pinned source versions) becomes the human-auditable artifact.

### Ship the mechanism, not the gem

Trust posture = reproducible build + property verifier, delivered as source. **Nothing
opaque in the chain**: source, manifest, generator, verifier all readable; the gem is
*derived and checked by readable code*. This is vault-manager's own model (clone → build →
shasum) but with a real property verifier where they have only `shasum`.

Two distinct checks, do not conflate:
- **Property verifier** (networked build side): byte-identical to pinned wallet, no excluded
  namespace reachable, every ceremony op runs.
- **Integrity hash** (at the air-gap crossing): "the gem that crossed is the one verified."

Must **pin the wallet + SDK versions** — the vault targets a released tag, never HEAD.

## Packaging / home

The mechanism is **strictly downstream of the wallet** (it consumes wallet + SDK, and the
wallet already depends on the SDK). That rules out the SDK repo/gem — a generator there would
have to reach *forward* to the wallet, its own consumer. That is the circular smell.

It is **build tooling**, so it ships in **no runtime gem**. It does not need to be its own
gem — a gem is something you keep and `require`; this you run once and discard. Hence
`bsv-vault-installer` feels wrong: it names a gem for a rake task.

**Home: a sibling package in the wallet monorepo, driven by a rake task.**

```
gem/bsv-vault/
├── README.md              # operator build instructions (lite manual)
├── Rakefile
├── manifest/
│   ├── wallet.yml         # pinned wallet version + exact file list
│   ├── sdk.yml            # pinned sdk version + exact file list
│   └── excluded.yml       # forbidden namespaces — the assurance claim in writing
├── glue/                  # the ONLY hand-written source (first-class, reviewed)
│   ├── lib/bsv/wallet/vault/human_chain_tracker.rb
│   ├── lib/bsv/wallet/vault/media_transfer.rb
│   └── bin/vault          # assembly: Store + Engine(calc) + glue, no Network
├── verify/
│   ├── byte_identity.rb
│   ├── boundary.rb
│   └── ceremony/          # exercises every vault op — sufficiency proof
├── tasks/vault.rake
└── build/                 # generated, gitignored — staging tree → .gem
```

SDK files come from the **resolved, pinned SDK gem on disk** (paths named in the manifest) —
no reach into the SDK repo, no vendoring, clean DAG (`sdk → wallet → vault-builder`).

**The mechanism we ship = this directory at a pinned wallet tag.** Operator clones
`wallet@vX`, runs `rake vault:build`, reads the attest record, shasums, carries it across.
No installer gem; the repo is the installer (vault-manager's model exactly).

### `rake vault:build` stages (each separately runnable, fails closed)

`resolve` (assert versions == manifest pins) → `extract` (copy exactly manifest files) →
`graft` (copy glue — the only non-subset code) → `byte_identity` (hash == source) →
`boundary` (each excluded namespace raises `NameError`; ceremony run leaves no forbidden
path in `$LOADED_FEATURES`) → `sufficiency` (ceremony green on the BUILT vault) →
`package` (reproducibility-denoised `gem build`) → `attest` (emit `.sha256` + build record).

Off the build path:
- `vault:manifest:derive` — **dev-time only** trace (Coverage/TracePoint) that *proposes* a
  manifest for human review. Never in the build.
- `vault:verify` — auditor re-runs `byte_identity`+`boundary`+`sufficiency` on a prebuilt gem.

## Sequencing — HLR + sub-issues

One HLR for the vault, spawning sub-issues; plans written per sub-issue, **spike-plan first**.

1. **Spike (gate).** Trace one `createAction`+sign; either prove the services layer never
   loads, or enumerate the leaks. Acceptance: a written verdict — *seams separate as-is* /
   *need cleanup X,Y* / *infeasible*. Everything downstream is gated on this. Cheap.
2. Manifest + deterministic generator (`resolve`/`extract`/`graft`).
3. Verifier (`byte_identity`/`boundary`/`sufficiency`; the separable `vault:verify`).
4. Glue (`HumanChainTracker`, `MediaTransfer`, `bin/vault` assembly).
5. Reproducible packaging + attestation.

Build-stage acceptance criteria are **only writable after the spike** — drafting them now
would commit to a shape the diagnostic might invalidate.

## Two things the mechanism HLR deliberately omits

1. **Seam-fix work is wallet work.** If the spike finds the Engine reaching for network code,
   fixing it is a wallet refactor — independently valuable, its own wallet-side issue, not
   buried in the vault HLR. The vault is partly a *forcing function* for seam cleanliness.
2. **The governance half is not code.** vault-manager's real substance was the operator
   manual (ceremony, build provenance, two-person rule). A software HLR won't capture it —
   it's a separate docs track and must not masquerade as falling out of the gem.
