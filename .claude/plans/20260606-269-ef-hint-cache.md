# Plan — #269: EF hint cache (in-process LRU + OMQ hint channel)

**Branch:** `feat/269-ef-hint-cache` off `feat/249-broadcast-network-boundary`
**Related:** #252 (provides the reconstruction floor), #271 (single submission codepath where the cache hooks)

## Context

Post-#252 every daemon broadcast pays `Store#find_action` + `Store#resolve_inputs_for_signing` — a JOIN across `inputs` × `outputs` × `actions` that scales with input count. The producer that signed the action already had a hydrated `Transaction` in memory; the daemon's reconstruction is *information re-fetching* across a process boundary, not work that needed to happen.

Primary motivation: **DB load shedding at scale.** Per-broadcast latency saving is a secondary nicety; the load reduction at the millions-TPS target is the actual win.

End-state architecture: walletd is the broadcaster (one or several long-running processes); CLI tools / API / UI are short-lived producer processes that sign and persist. Two channels between them:

| Channel | Authority | On failure |
|---|---|---|
| DB write (`Store#sign_action`) | Authoritative — action exists, queued for broadcast | Producer raises; create_action rolls back |
| OMQ hint push | Opportunistic — gives daemon the hydrated Transaction | Silently no-op; daemon reconstructs via #252 |

Correctness rides on the DB. Performance rides on the hint.

## Sub-pieces

Three independent, each independently testable. Single PR or split — call at PR time.

### 1. `Engine::EFCache` (in-process LRU on the daemon)

- File: `gem/bsv-wallet/lib/bsv/wallet/engine/ef_cache.rb`.
- Interface: `#get(action_id) → Transaction | nil`, `#put(action_id, tx)`, `#evict(action_id)`, `#size`, `#capacity`.
- Implementation: Ruby Hash (insertion-ordered, doubles as MRU tracker) + `Mutex` for fiber/thread safety + bounded LRU eviction (`shift while size > capacity`).
- Capacity: default 1000, env `BSV_WALLET_EF_CACHE_SIZE`.
- ~30 lines of code, zero deps.
- Default constructor in `Engine::Broadcast` so non-daemon callers get a real cache too (low-overhead; no opt-out needed).

### 2. OMQ hint receiver (daemon side)

- New fiber in `BSV::Wallet::Daemon#run!` alongside the existing pull workers.
- Binds `OMQ::PULL` on a socket path (default `ipc:///var/run/walletd-<wallet>/ef-hints.sock` or `inproc://ef-hints` for in-process / test runs).
- Loop: `pull.receive` → `Marshal.load` → `{ action_id:, ef_bytes: }` → `Transaction.from_binary(ef_bytes)` → `@cache.put(action_id, tx)`.
- Failure modes: malformed message → log + skip (don't crash the fiber); same pattern as `statuses_pull!`.

### 3. OMQ hint publisher (producer side)

- New module on `Engine`: `EFHintPublisher` (or methods on Engine directly — call at PR time).
- After `Store#sign_action` commits inside `Engine#create_action`, push `Marshal.dump({ action_id: id, ef_bytes: tx.to_binary })` to the same socket.
- Send is non-blocking (`OMQ::PUSH#send_nowait` or equivalent; if buffer is full or daemon isn't listening, drop the hint).
- Socket opened lazily on first publish; reused for process lifetime.
- Opt-in via env (`BSV_WALLET_EF_HINTS_SOCKET=ipc://...`). When unset, publisher is a no-op (default behaviour unchanged).
- One-shot CLI tools (`bin/create_action`) get the env from `.env` / shell; they don't need to know about it directly.

### 4. Cache hit on the broadcast path

- `Engine::Broadcast#hydrated_transaction_for` gains a one-line short-circuit at the top:
  ```ruby
  cached = @cache.get(action[:id]) and return cached
  ```
- Existing parse + DB-JOIN path stays unchanged as the miss path.

### 5. Eviction on terminal outcomes

- `Engine::Broadcast#handle_submit_success` evicts after `record_broadcast_result` (the action is on-chain or in mempool; the cached Transaction is dead weight).
- `Engine::Broadcast#handle_submit_terminal` evicts after `reject_action` (the cascade already deleted the action; the cache entry would dangle).
- Backpressure 503: keep the entry (the broadcast will be retried next cycle; cache hit still useful).

## Specs

- `spec/bsv/wallet/engine/ef_cache_spec.rb`: get/put/evict roundtrip, LRU eviction at capacity, fiber-safety (parallel Async puts/gets), capacity = 0 edge case.
- `spec/bsv/wallet/engine/broadcast_spec.rb`: 
  - `hydrated_transaction_for` hits cache → no `find_action` / `resolve_inputs_for_signing` calls.
  - `hydrated_transaction_for` miss → falls through to today's path (existing specs unchanged).
  - terminal success evicts; terminal reject evicts; 503 does not evict.
- `spec/bsv/wallet/daemon_spec.rb`: hint receiver fiber pulls a message, populates the cache.
- `spec/bsv/wallet/engine_spec.rb`: `create_action` calls the publisher after sign when configured; no-op when not configured.
- Integration: `walletd_events_spec` extended (or new spec) — push a hint via the publisher, verify daemon's submit doesn't query the DB for source data.

## Live verification

1. Run walletd against alice with `BSV_WALLET_EF_HINTS_SOCKET=ipc:///tmp/alice-hints.sock`.
2. `BSV_WALLET_EF_HINTS_SOCKET=ipc:///tmp/alice-hints.sock bin/create_action alice --inline --description "269 cache hit"`.
3. Daemon log shows broadcast_submission with no `resolve_inputs_for_signing` debug line (or whatever a tracing hook surfaces).
4. tx mines, no behavioural difference vs without the hint.

## Risks

- **Stale cache entries.** If `record_broadcast_result` is called via a path that doesn't go through `handle_submit_success` (e.g. SSE applicator updating tx_status post-discovery), the entry might not be evicted. LRU is the backstop. Worth a spec.
- **OMQ socket lifecycle.** ipc:// sockets are files on disk; need a clean unlink at daemon shutdown (and tolerate stale sockets on startup — bind with `IPV6_V6ONLY` / overwrite semantics, depends on OMQ binding behaviour).
- **Default cache enabled non-daemon contexts.** A bounded LRU costs nothing per-call when empty, so default-on is fine. But every Engine spec that builds an engine now gets a cache; ensure no spec asserts on cache absence (none should, since the cache is internal).
- **Single-host scope.** ipc:// is single-host; multi-host walletd requires tcp:// + auth. Out of scope; the publisher/receiver interface unchanged when that lands.

## Out of scope

- Cross-host via Redis / tcp://. Different PR.
- Size-based filtering ("don't push tiny txs"). Premature.
- Cache for incoming BEEF / proofs. Different lifecycle.
- Persistent EF column on `actions`. Architecturally rejected in #252.
